# 2026-04-08 — v0.1.0 implementation and beta test

Session: kväll. End-to-end från tom projektmapp till testad v0.1.0 beta på GitHub.

## Vad som gjordes

Byggde hela `mcp-refchecker` från grunden: MCP-server som wrappar [academic-refchecker](https://github.com/markrussinovich/refchecker) och exponerar verktyget `verify_citation` för Claude. Målet är att låta Claude verifiera akademiska citeringar mot Semantic Scholar / OpenAlex / Crossref för att fånga hallucinationer innan de hamnar i färdigt arbete.

Sluttad med:
- Paketstruktur `mcp_refchecker/` + `pyproject.toml`
- FastMCP-server (~220 rader) med verify_citation
- `.gitignore`, `README.md`, `CLAUDE.md`
- GitHub-repo på https://github.com/JonasBaath/mcp-refchecker (privat, tag v0.1.0 pre-release)
- 10/10 godkända betatest

## Arkitektoniska beslut

**Importera paketet direkt, inte wrappa REST/CLI.** Alternativ 3 från research-fasen (2026-04-08 morgon). Ger en enda pip-dependency och rent API mot `ArxivReferenceChecker`-klassen.

**Singleton-checker + `asyncio.Lock`.** `ArxivReferenceChecker.__init__` bygger hela hybrid-checkern, eventuell LLM och web-sökare — det är dyrt. Initieras lat vid första tool-anropet och cachas.

**`asyncio.to_thread(checker.verify_reference, ...)`.** `verify_reference` är synkron och gör HTTP-anrop, så den måste av async event-loopen. FastMCP kräver async tool-funktioner.

**`llm_config={"disabled": True}`.** academic-refchecker stödjer även LLM-baserad hallucinationsdetektion, men det kräver egen API-nyckel och är ortogonal mot vårt syfte. Vi vill ha källverifiering, inte LLM.

**`types.SimpleNamespace`-stub för `source_paper`.** `verify_reference` kräver ett `source_paper`-objekt med `title`/`authors`/`published.year`/`canonical_url`/`external_paper_id`/`venue`, men använder det bara för loggning i vår use case. En stub räcker och slipper filsystemberoenden.

## Buggar upptäckta och fixade

### 1. `KeyError: 'raw_text'`
`verify_reference_standard` anropar `logger.debug(f"Raw text: {reference['raw_text']}")` vid fallback-paths, utan att kontrollera om fältet finns. Vi byggde bara `{"title", "authors", "year", ...}` i `_build_reference`. Fix: bygg `raw_text` från tillgängliga fält (`authors (year) title`) som en syntetisk referenssträng.

### 2. `verified` var alltid `false` för papers som hittades
Första implementationen satte `verified: errors is None`. Men `academic-refchecker` returnerar ofta warnings även för korrekt-hittade papers (t.ex. "arXiv preprint citerat istället för publicerad version"). Fix: separera `error_type` från `warning_type`/`info_type`, bara `error_type`-poster blockerar verifiering.

### 3. "Missing field"-fel för incomplete input
När användaren bara angav titel utan år/venue, returnerade refchecker `error_type: "year"` med `"Year missing"`. Det sätter `verified: false` även när paperet hittades perfekt. Fix: om paperet hittades och felet handlar om att ett fält saknas (inte är fel), demotera till warning.

### 4. Year-mismatch markeras som warning, inte error (inkonsekvens i refchecker)
"Rätt titel, fel år" returnerade `warning_type: "year"` med "Year mismatch: cited 2019 vs actual 2017". Men "Rätt titel, fel författare" returnerade `error_type: "author"` med samma typ av mismatch. Fix: promotera `warning_type: year/author/venue` med "mismatch" i details till hårda fel. Skippa version-suffixade varianter (`(v6 vs v7)`) och arxiv-preprint-venue-varningar.

## Fuzzy fallback-sagan

Målet: fånga titlar som refchecker avfärdar för strikt — t.ex. "Atention Is All You Need" (ett bokstav saknas). Påbörjade med antagandet att enkelt fuzzy-matchning räcker.

**Försök 1: Semantic Scholar `/paper/search/match`.**
Returnerade 404 för vaga queries. Skiftade till `/paper/search` med top-5 och fuzzywuzzy.ratio-sortering.

**Försök 2: Semantic Scholar `/paper/search` med fuzzywuzzy.**
Fungerade när jag testade utanför scriptet — men i praktiken fick jag HTTP 429 (rate-limit) direkt. S2:s fria tier delar en global pool mellan alla oautentiserade användare och är mycket strikt.

**Försök 3: OpenAlex `/works?search=...`.**
Inget rate-limit, men... returnerade **0 kandidater** för BERT med 2 typos, och fem helt orelaterade papers för "Atention" (ratio 30–40%). OpenAlex gör strikt token-sökning — ett felstavat ord finns inte i indexet, punkt slut. Dessutom returnerade OpenAlex fel metadata för "Attention Is All You Need" (år 2025, obskur placeholder-DOI) — verkar finnas en skuggentry av paperet.

**Försök 4: Crossref `/works?query.title=...`.**
Bättre än OpenAlex för exakta titlar. Men för "Atention" returnerade Crossref fem Beatles-låtar om "All You Need" — inte en enda hade "Attention" i titeln. Samma grundproblem: Crossref dropparade det felstavade ordet ur sökningen, och då fanns inte rätt paper i resultatlistan alls.

### Insikt

**Ingen fri akademisk API har fuzzy titelmatchning som klarar stavfel.** Alla (S2, OpenAlex, Crossref) gör keyword/token-baserad sökning. Så fort ett distinktivt ord är felstavat finns det inte i indexet, och det riktiga paperet dyker inte upp i resultaten oavsett hur man post-processar dem.

Att lösa det skulle kräva antingen:
1. Semantic embeddings via paid API (OpenAI, Voyage) — matchar på mening, inte stavning
2. Full-text fuzzy search engine (Elasticsearch med fuzzy query) — inte exponerad av någon fri akademisk datakälla

**Beslut:** Acceptera begränsningen för v0.1.0. Dokumentera ärligt i README. Fuzzy fallback (mot Crossref) fångar fortfarande värdefulla stilistiska variationer (skiftläge, punktuation, ordföljd, små omformuleringar), bara inte riktiga stavfel i kärnord.
